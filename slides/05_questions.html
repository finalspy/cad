Questions ?

Quelles sont vos questions sur les architectures logicielles ?

Quelles sont vos réflexions sur les architectures monolithiques et distribuées ?

🧱 1. Monolithe vs Mainframe

  • Un monolithe, c'est forcément une vieille application sur un mainframe ?
  • Un monolithe ne peut pas être moderne ?
  • Si une application est monolithique, cela veut dire qu'elle ne peut tourner que sur un seul ordinateur ?
  • Non, un monolithe peut être développé aujourd'hui avec des technos modernes. Ce n'est pas lié aux mainframes.
  • Un monolithe peut être propre, performant et maintenable. Ce n'est pas un synonyme d'application obsolète.
  • Une application monolithique peut être déployée sur plusieurs machines, même si elle reste un seul bloc logiciel.

🧩 2. Monolithe vs Microservices

  • Si une application est divisée en plusieurs fichiers ou modules, est-ce qu'elle n'est plus un monolithe ?
  • Est-ce qu'on est obligé d'avoir plusieurs dépôts de code pour faire une architecture microservices ?
  • Est-ce que les microservices sont toujours plus performants qu'un monolithe ?
  • Est-ce que c'est mieux de toujours découper son application en microservices dès le début ?
  • Si je peux redémarrer un service sans toucher au reste de l'application, est-ce que c'est forcément un microservice ?
  • Non, un monolithe peut être bien structuré en modules sans être distribué ou microservices.
  • Non, on peut avoir tous les services dans un monorepo. Ce n'est pas une obligation technique.
  • Pas toujours : plus de complexité, de communication réseau, de latence possible. Ce n'est pas automatique.
  • Pas nécessairement : pour un petit projet ou un MVP, un monolithe est souvent plus simple à gérer.
  • Non : ce critère est typique, mais ce n'est pas suffisant pour définir un microservice.

🧪 3. Microservices vs SOA

  • Si mes services communiquent entre eux, est-ce qu'on est automatiquement dans une architecture SOA ?
  • Peut-on avoir une architecture microservices sans architecture SOA ?
  • Est-ce que microservices et SOA, c'est la même chose ?
  • Non, la communication entre services ne suffit pas à définir SOA. L'approche, les contrats et la gouvernance comptent aussi.
  • Oui, les microservices sont une évolution de SOA mais avec une granularité plus fine, des déploiements indépendants, etc.
  • Non, ce sont deux concepts différents, même s'ils sont liés. Les microservices sont une approche plus légère et décentralisée.

☁️ 4. Microservices et déploiement

  • Est-ce qu'une architecture microservices doit forcément être déployée sur le cloud ?
  • Peut-on faire une architecture microservices sur un seul serveur ?
  • Est-ce qu'une application est "distribuée" uniquement si elle tourne sur plusieurs serveurs ?
  • Non, on peut faire du microservices en local ou on-premise. Le cloud n'est pas obligatoire.
  • Oui, on peut faire tourner plusieurs services sur un même serveur physique, voire dans le même container au début.
  • Non, plusieurs processus interconnectés suffisent, même sur une seule machine.

🧮 5. MVC, n-tiers et structure de code

  • Une application avec une architecture en couches (MVC ou n-tiers) est-elle forcément distribuée ?
  • Le modèle MVC ne s'utilise que dans les architectures n-tiers ?
  • Si j'utilise un framework moderne, est-ce que je fais automatiquement une architecture n-tiers ?
  • Est-ce qu'une application MVC est forcément web ?
  • Est-ce que le fait d'avoir une partie “vue” dans le navigateur et une “logique” sur un serveur rend l'application distribuée ?
  • Non, MVC et n-tiers sont des patterns de structuration logique, pas de distribution physique.
  • Non, on peut appliquer MVC en local, dans une appli bureau par exemple.
  • Non, le framework ne dicte pas l'architecture physique, juste la structure logique.
  • Non, MVC est aussi utilisé pour des applis desktop ou mobiles.
  • Oui, techniquement, c'est une forme simple d'architecture distribuée (client-serveur).

🌍 6. Application distribuée ou pas ?

  • Une application distribuée, c'est juste une appli qui communique avec un serveur ?
  • Est-ce qu'une application distribuée tourne forcément sur plusieurs machines ?
  • Est-ce que “application distribuée” veut dire “microservices” ?
  • Est-ce que l'utilisateur d'une application distribuée interagit avec plusieurs applications différentes ?
  • En partie, oui. C'est le cas typique du client-serveur. Mais ce n'est pas l'unique forme de distribution.
  • Non, plusieurs processus interconnectés suffisent, même sur une seule machine.
  • Non, une application distribuée peut être un monolithe déployé sur plusieurs nœuds, ou une appli client-serveur classique.
  • Non, elle peut percevoir un tout cohérent même si l'architecture est distribuée en interne.

👥 7. Organisation des équipes et architecture

  • Si plusieurs équipes travaillent sur le même projet, est-ce qu'il faut forcément passer à une architecture microservices ?
  • Avoir plusieurs développeurs, ça veut dire qu'on ne peut pas rester en monolithe ?
  • Non, on peut très bien organiser le travail sur un monolithe en modules ou sous-projets bien séparés.
  • Non, ce n'est pas le nombre de développeurs qui dicte l'architecture, mais les besoins métiers, la scalabilité, la maintenabilité…

🔧 8. Outils et technologies

  • Est-ce qu'on doit utiliser des outils spécifiques pour faire du microservices ?
  • Est-ce que les microservices sont réservés à certaines technologies ?
  • Est-ce que le cloud est obligatoire pour faire du microservices ?
  • Non, on peut utiliser des outils variés, mais certains facilitent la gestion des microservices.
  • Non, on peut faire du microservices avec n'importe quelle technologie, même des langages moins courants.
  • Non, le cloud est un choix, pas une obligation. On peut faire du microservices sur site.

🔒 9. Sécurité

  • Est-ce que les microservices sont plus sécurisés qu'un monolithe ?
  • Est-ce que la sécurité est plus complexe dans une architecture distribuée ?
  • Est-ce que le fait d'avoir plusieurs services rend l'application moins sécurisée ?
  • Pas forcément, la sécurité dépend de la mise en œuvre, pas de l'architecture.
  • Oui, la gestion des accès, des communications et des données est plus complexe.
  • Non, mais il faut bien gérer les accès et les communications entre services.
