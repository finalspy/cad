🧩 Architecture et Design

Ces patterns aident à structurer votre code et votre base de données autour de la logique métier, tout en favorisant l'indépendance des services.

Domain Driven Design (DDD)

  • Ubiquitous Language, Bounded Context
  • Entities, Value Objects, Aggregates
  • Domain Services, Domain Events, Repositories
La DDD permet de faire correspondre la structure du code à celle du métier. On isole les responsabilités métier dans des contextes bien délimités.

Database per Domain

  • Chaque service a sa propre base
  • Pas de couplage via la base de données
Cela permet une indépendance de déploiement, et évite les verrouillages entre équipes ou modules.

CQRS & Event Sourcing

  • CQRS : séparation écriture/lecture
  • Event Sourcing : reconstruction via les événements
Ces patterns apportent traçabilité et performance, mais compliquent le modèle conceptuel.

Saga Pattern

  • Choreography : réaction en chaîne sans chef d'orchestre
  • Orchestration : coordination centralisée
Permet de gérer des transactions longues et distribuées sans rollback global.

Couplage faible

  • Isolation des services
  • Utilisation d'événements ou d'API explicites
Un service doit pouvoir évoluer ou tomber sans mettre à mal les autres.
